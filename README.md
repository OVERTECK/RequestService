Эндпоинты, Для кого, Назначение:
    POST /api/requests - Сотрудник - Подать заявку
    GET /api/requests/mine - Сотрудник - Свои заявки и их статусы
    GET /api/requests?status= - Бухгалтер - Очередь заявок, фильтр по статусу
    GET /api/requests/{id} - Бухгалтер - Детали заявки
    PATCH /api/requests/{id}/status - Бухгалтер - Сменить статус
    GET /api/requests/{id}/history - Оба - История изменений статуса

Вопросы:
    1. Какое архитектурное решение вы считаете самым рискованным и почему?
    
    Выбор SQLite и хранение состояния в одном процессе. Это идеально закрывает требование
    «запуск без доп. компонент» и ускоряет разработку, но SQLite плохо переносит конкурентную
    запись и не масштабируется горизонтально. Под реальной нагрузкой это первое узкое место.

    2. Где вы сознательно оставили технический долг? Как бы его закрыли при наличии времени?
    
    1) Нет пагинации в очереди заявок — на больших объёмах ответ распухнет. Добавил бы
    пагинацию и индексы.
    2) Авторизация через заголовки (X-Role/X-User-Id) вместо реальной аутентификации.
    Закрыл бы через JWT с проверкой ролей в middleware.
    
    3. Что бы вы изменили, если бы знали, что сервис станет высоконагруженным?

    1) Перешёл бы с SQLite на PostgreSQL
    2) Добавил бы пагинацию, кеширование справочников.
    3) Идемпотентность вынес бы в Redis с TTL вместо строки в основной таблице.
